# Bestemshe Solver 2.0 — Compute & Memory Design Spec

**Date:** 2026-06-24
**Status:** Approved for planning
**Scope:** Two-phase upgrade of the retrograde solver. Phase 1 raises solve throughput
(target ≈ 20 M states/s, from ≈ 5.8 M). Phase 2 removes the 16 GB RAM ceiling so the
final descent to M=0 is feasible on the host machine.

---

## 1. Context & goals

The solver currently packs 4 states per byte in a `std::vector<std::atomic<uint8_t>>`
and writes each state with a `compare_exchange_weak` (CAS) loop, reads with a shift/mask
atomic-acquire load, and walks the whole layer with manual static OpenMP chunking. The
retrograde value iteration (`solve_layer_lock_free`) is correct and already uses the
O(1) board odometer (`StateIndex::AdvanceBoard`) from the prior change set.

Two goals:

1. **Throughput** — remove the per-write CAS, evaluate winning moves earlier, and
   load-balance the sweep. (Phase 1, benefits the current working range M≈16–48.)
2. **Memory** — at low M the layer state space explodes (`C(59-M,9)` grows fast), so the
   whole-layer array plus the InferenceEngine's preload of *all* higher layers exceeds
   16 GB. Solve one symmetric kazan pair at a time, preloading only the higher
   micro-layers that pair can actually reach. (Phase 2, enables M ≤ ~14 down to 0.)

### In scope
- **A1** — 1-byte-per-state solve store with relaxed (CAS-free) access; adaptive 2-bit
  fallback when the 1-byte array will not fit the RAM budget.
- **A2** — landing-pit move ordering (try opponent-side-landing moves first).
- **A3** — dynamic-scheduled 64 K-state block odometer.
- **Phase 2** — pair-by-pair solving with selective (reachable-only) preloading; 4-bit
  store gated to the single oversized M=2 pair.

### Out of scope (with rationale)
- **B4 (XOR-with-heuristic encoding) — dropped.** Measured on solved layers M=24 (12,12)
  and M=20 (10,10): the best cheap heuristic only matches the class base rate (≈63–65%),
  board-material heuristics fall *below* base rate, and XOR-encoding makes every ZSTD plane
  **larger** (win +0.8%, draw +4–5%). Structural reason: within a micro-layer file
  `(K_self,K_opp)` are constant, so kazan heuristics are constant predictions; the only
  intra-file signal is the board, which is anti-predictive here.
- **B5 (bit-planar transposition) — already implemented.** `write_raw_monoliths` emits
  separate `win_bits`/`draw_bits` planes; `Splitter` writes `layer_k1_k2_win.raw` and
  `..._draw.raw`; each is ZSTD-compressed independently. The 2-bit interleaving exists only
  in the transient solve array, which is never compressed.
- **mmap overflow — rejected.** Scattered value-iteration access to paged memory thrashes
  the SSD (TLB misses + page faults dominate; UMA bandwidth does not help random faults),
  and the proposed flag expression `MAP_ANONYMOUS | exists(...) ? A : B` drops
  `MAP_ANONYMOUS` (the `|` binds tighter than `?:`).

## 2. Phase 1 — throughput

### 2.1 A1 — store abstraction (CAS-free 1-byte, adaptive 2-bit fallback)

**Problem.** Every `WriteState2Bit` is a CAS loop because 4 states share a byte; even
uncontended, a CAS is a locked, full-barrier op (~tens of cycles) executed once per proven
state, and each of the ≤5 per-state move lookups pays an atomic-acquire load + shift/mask.

**Design.** Introduce a store concept with two concrete implementations, selected once per
solve and dispatched through a template so hot-loop accesses stay inlined (no virtual call):

```cpp
// New: StateStore.h
enum class StoreMode { ONE_BYTE, TWO_BIT, NIBBLE };   // NIBBLE used only by the M=2 pairwise path

struct ByteStore {                       // ONE_BYTE: 1 atomic<uint8_t> per state
    std::vector<std::atomic<uint8_t>> v;
    explicit ByteStore(uint64_t n) : v(n) {
        for (auto& x : v) x.store(0, std::memory_order_relaxed);
    }
    inline GameValue read(uint64_t i) const {
        return static_cast<GameValue>(v[i].load(std::memory_order_relaxed));
    }
    inline void write(uint64_t i, GameValue g) {
        v[i].store(static_cast<uint8_t>(g), std::memory_order_relaxed);   // no CAS
    }
};

struct PackedStore {                     // TWO_BIT: current 4-states/byte + CAS
    std::vector<std::atomic<uint8_t>> v;
    explicit PackedStore(uint64_t n) : v((n + 3) / 4) { /* zero-init */ }
    inline GameValue read(uint64_t i) const;        // shift/mask, acquire load
    inline void write(uint64_t i, GameValue g);     // CAS loop (today's WriteState2Bit)
};
```

`solve_layer_lock_free` becomes a thin dispatcher over a templated body:

```cpp
template <class Store> void RetrogradeSolver::solve_impl(Store& store);   // the sweep loop

void RetrogradeSolver::solve_layer_lock_free() {
    if (choose_store_mode() == StoreMode::ONE_BYTE) { ByteStore s(num_states);  solve_impl(s); pack_and_write(s); }
    else                                            { PackedStore s(num_states); solve_impl(s); pack_and_write(s); }
}
```

`pack_and_write` is the existing `write_raw_monoliths` logic, reading via `store.read(i)`
and emitting the unchanged win/draw bit-planes — so on-disk output is **byte-identical**
regardless of store mode.

**Mode decision.** `choose_store_mode()` compares the 1-byte footprint against a budget:

```cpp
constexpr uint64_t RAM_BUDGET = 14ull << 30;   // leave headroom on a 16 GB host
StoreMode choose_store_mode() const {
    uint64_t one_byte = num_states;                       // bytes
    uint64_t preload  = estimate_preload_bytes(layer_M);  // sum of higher bit-planes (whole-layer path)
    return (one_byte + preload <= RAM_BUDGET) ? StoreMode::ONE_BYTE : StoreMode::TWO_BIT;
}
```

`estimate_preload_bytes(M)` = Σ over the layers preloaded of `GetLayerSize(M')/4` (2 bits per
state across win+draw planes). For the whole-layer path this is M'=M+2..48. Below the M
where even the 2-bit whole-layer + full preload exceeds budget, the pipeline switches to
Phase 2 (pairwise) — see §3.

**Correctness.** Writes are monotone (UNKNOWN→WIN/LOSS/DRAW only); within a sweep each
index is written by exactly one thread; cross-thread reads are relaxed loads. The implicit
OpenMP barrier between sweeps publishes all writes before the next sweep reads them, and the
convergence counter (`states_proven_this_iter`, atomic) guarantees another sweep whenever any
state changed. Relaxed ordering is therefore sufficient; a stale within-sweep read only
defers a resolution to a later sweep.

### 2.2 A2 — landing-pit move ordering

**Problem.** Moves are evaluated `0,1,2,3,4`. The win cutoff (`target_val == LOSS ⇒ WIN,
break`) fires only after the winning move is reached; a late winning capture wastes the
earlier evaluations.

**Landing pit (cheap, no full sow).** For pit `i` with `c = board[i]` stones, the last
sown stone lands at:

```
land(i) = (c == 0) ? NONE
        : (c == 1) ? (i + 1)      % 10
        :            (i + c - 1)  % 10     // c stones: pit i refilled to 1, then c-1 sown
```

A capture can only occur when `5 ≤ land(i) ≤ 9` (opponent side). Captures are the forcing
moves most likely to prove a WIN, so we evaluate opponent-side-landing moves first.

**Loop structure.** Build a 5-entry order once per state by a cheap partition (no sort):

```cpp
int order[5]; int n = 0;
for (int i = 0; i < 5; ++i) if (board[i] && lands_opponent(i, board[i])) order[n++] = i; // captures-likely first
for (int i = 0; i < 5; ++i) if (board[i] && !lands_opponent(i, board[i])) order[n++] = i;
for (int k = 0; k < n; ++k) { int move = order[k]; /* existing evaluation + cutoff */ }
```

The win/loss/draw logic is order-independent (`proved_win` = ∃ LOSS-target;
`all_losses_for_me` = ∀ WIN-target), so reordering never changes results — only when the
cutoff fires. **This is the lowest-confidence item** (gain measured ~10–20% on WIN states
only, none on LOSS/DRAW states); it ships only if a before/after benchmark shows net gain,
else it is reverted.

### 2.3 A3 — dynamic-scheduled block odometer

**Problem.** Manual static chunking idles threads in late sweeps, where the few remaining
UNKNOWN states (deep draws) cluster unevenly across the index range.

**Design.** Replace the manual `#pragma omp parallel { per-thread begin/end }` with dynamic
64 K-state blocks; each block seeds the board once and odometers the rest:

```cpp
constexpr uint64_t BLOCK = 65536;
uint64_t num_blocks = (num_states + BLOCK - 1) / BLOCK;
#pragma omp parallel for schedule(dynamic, 1)
for (uint64_t b = 0; b < num_blocks; ++b) {
    uint64_t begin = b * BLOCK, end = std::min(num_states, begin + BLOCK);
    SolverState s = UnindexState(begin, layer_M);          // one decode per 64 K states
    for (uint64_t i = begin; i < end; ++i) {
        /* read store.read(i); if UNKNOWN evaluate moves (A2 order); write store.write(i,...) */
        if (i + 1 < end) advance_solver_state(s, R);        // O(1) odometer within block
    }
}
```

Blocks are independent (each re-seeds via `UnindexState`), so dynamic stealing is safe and
keeps all cores busy. Applies to both `solve_impl` and `verify_layer_consistency`.

### 2.4 Phase 1 tests
- **Equivalence:** solve a mid layer (e.g. M=40) with `--solve` before/after; win+draw
  monoliths must be **byte-identical** to the pre-Phase-1 binary (covers A1, A2, A3 together).
- **Store-mode equivalence:** force ONE_BYTE and TWO_BIT on the same layer; outputs identical.
- **`--selftest`** still passes (bijection + odometer) on M=48/46/44/40/30.
- **Throughput:** print states/s from existing `[BENCHMARK]` timing; compare before/after.

## 3. Phase 2 — pair-by-pair solving + selective preloading

### 3.1 The independence theorem

Within layer M, classify states by their kazan pair `(K_self, K_opp)`. From a state in
micro-layer `(K1, K2)` (M = K1+K2):

- **Non-capture move:** kazans unchanged, board flips, active player swaps ⇒ result is in
  `(K2, K1)`, **same M**.
- **Capture move (c stones, c even ≥ 2):** `K_self += c` then flip ⇒ result is in a layer
  `M' = M + c > M`, already solved and on disk.

Therefore micro-layer `(K1,K2)` reads only `(K2,K1)` within layer M plus on-disk higher
layers — **never any other pair `(Ka,Kb)` in layer M**. Layer M partitions into independent
subgraphs `{(K1,K2),(K2,K1)}`. (Verified against `ExecuteMoveAndFlip` + the solver's
transition handling.) Each pair can be solved in isolation in RAM, then written to disk,
before the next pair is touched.

### 3.2 Pairwise solver — local indexing & loop

New entry point `solve_layer_pairwise()` loops `K1 = GetMinK(M) .. floor(M/2)`, `K2 = M-K1`:

- **Store size:** `slices * b_count`, `slices = (K1 == K2) ? 1 : 2`, `b_count = C(50-M+9, 9)`.
  Slice 0 = `(K_self=K1,K_opp=K2)`, slice 1 = `(K_self=K2,K_opp=K1)`.
- **Board-only index** (new helper, factored from `IndexState`/`UnindexState`):

```cpp
static uint64_t StateIndex::IndexBoard(const uint8_t b[10]);   // returns I_B only (no I_K term)
```

- **Same-layer (non-capture) target → local store index:**

```cpp
int slice = (next_s.K_self == K1) ? 0 : 1;          // K1==K2 ⇒ always 0
uint64_t local = static_cast<uint64_t>(slice) * b_count + StateIndex::IndexBoard(next_s.board);
GameValue tv = store.read(local);
```

- **Capture target:** unchanged — `inference->query_state(M', next_s.K_opp, IndexState(next_s))`.
- **Terminal:** unchanged — `empties_opponent || next_s.K_opp >= 26 ⇒ LOSS`.
- **Sweep:** the §2.3 dynamic block odometer runs over `[0, slices*b_count)`; for slice `sl`
  the seed is `UnindexState(I_K(sl)*b_count + block_begin, M)` where
  `I_K(0) = (K1-GetMinK(M))/2`, `I_K(1) = (K2-GetMinK(M))/2`.
- **Convergence:** identical fixpoint loop (sweep until `states_proven == 0`, then remaining
  UNKNOWN → DRAW via `finalize_draws` over the local store).
- **Output:** after convergence, down-pack each slice directly to
  `layer_{k1}_{k2}_win.raw` / `..._draw.raw` (and the mirror for slice 1). This bypasses the
  monolith + `Splitter` step entirely for the pairwise path. Free the store + preloaded
  files before the next pair.

### 3.3 Selective preloading — mathematical derivation

A capture from orientation `(K_self, K_opp)` produces, after the flip, the queried state
`(K_self'=K_opp, K_opp'=K_self+c)` in layer `M' = M + c`. The InferenceEngine keys a
micro-layer file as `layer_{k1'}_{k2'}` with `k2' = K_opp'` and `k1' = M' - k2' = K_opp`.
Captures are queried only while non-winning, i.e. `K_self + c ≤ 24` (else `K_opp' ≥ 26`
is the terminal LOSS branch, no query). With `c` even ≥ 2, the reachable files are:

- **From `(K1,K2)`** (K_self=K1): `k1' = K2`, `k2' = K1 + c ∈ {K1+2, K1+4, …, 24}` ⇒
  files `{ layer_{K2}_{j} : j even, K1 < j ≤ 24 }`.
- **From `(K2,K1)`** (K_self=K2): `k1' = K1`, `k2' = K2 + c ∈ {K2+2, …, 24}` ⇒
  files `{ layer_{K1}_{j} : j even, K2 < j ≤ 24 }`.

So the **selective preload set** for pair `{(K1,K2),(K2,K1)}` is

```
Reach(K1,K2) = { (K2, j) : j even, K1 < j ≤ 24 }  ∪  { (K1, j) : j even, K2 < j ≤ 24 }
```

each `(k1',k2')` residing in layer `M' = k1'+k2'`. (`k1'` is always a valid K_self in `M'`:
`GetMinK(M') = max(0, M'-24) ≤ k1'` reduces to `k2' ≤ 24` ✓.) For `K1==K2` the two sets
coincide. *Example:* pair `(2,4)` reaches `layer_4_{4,6,…,24}` ∪ `layer_2_{6,8,…,24}` —
a small constant set versus the entire M=8..48 cone that `preload_all_layers` loads today.

New InferenceEngine API:

```cpp
void InferenceEngine::preload_pair(uint8_t K1, uint8_t K2);   // loads exactly Reach(K1,K2)
```

implemented by enumerating `Reach(K1,K2)` and calling the existing
`preload_uncompressed_layer(M', k2')` per file. Peak preload RAM is bounded by the size of
this set, not the whole higher cone.

### 3.4 4-bit store for the M=2 pair

The single M=2 pair `{(0,2),(2,0)}` is `2 × C(57,9) ≈ 17.7 GB` at 1 byte — the only pair
over budget. Add a `NibbleStore` (2 states/byte): with byte-aligned block boundaries (block
size a multiple of 2 states) no two threads write the same byte, so writes are relaxed
RMW on an `atomic<uint8_t>` with no CAS; cross-thread reads are relaxed loads. Footprint
8.86 GB. The pairwise solver picks its own store — `ByteStore` for every pair by default,
`NibbleStore` only when `M == 2` — independently of the whole-layer `choose_store_mode`.

### 3.5 Pipeline integration

- New CLI mode `--solve-pairwise <M> <manifest>` driving `solve_layer_pairwise()` and
  writing `layer_k1_k2_*.raw` directly.
- `pipeline.cpp` selects per layer: `--solve` (whole-layer, Phase 1) while the full preload
  fits the budget; `--solve-pairwise` below an empirically tuned threshold M*. The pairwise
  branch **skips `--split`** (raws are produced directly) and goes straight to `--compress`.
- `--solve` (whole-layer) and `Splitter` remain for the high-M range and as the reference
  implementation for equivalence testing.

### 3.6 Phase 2 tests
- **Equivalence vs reference:** for a mid layer solvable both ways (e.g. M=24), compare
  `--solve` + `--split` outputs against `--solve-pairwise` outputs per micro-layer —
  **byte-identical** `.raw` files.
- **Selective-preload sufficiency:** `--verify <M>` after a pairwise solve must report
  100% self-consistent (a missing reachable file would surface as a value discrepancy).
- **Reachability unit test:** assert `Reach(K1,K2)` exactly equals the set of
  `(k1',k2')` produced by enumerating every legal capture from every state of a small pair.
- **Memory bound:** `/usr/bin/time -l` peak RSS for a low-M pairwise solve stays under budget.

## 5. Risks & open questions
- **A2 may not pay for itself** — ordering overhead vs cutoff savings is workload-dependent;
  gated behind a benchmark, revert if not net-positive.
- **Threshold M\*** between whole-layer and pairwise is empirical; tune from measured RSS.
- **Template bloat** — two/three store instantiations of the sweep body increase binary
  size marginally; acceptable for the inlining win.
- **Pairwise output path** diverges from monolith/Splitter — equivalence test (§3.6) is the
  guard; keep whole-layer path as the reference.
